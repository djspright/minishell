# MINISHELL プロジェクト評価シート

## はじめに

評価プロセス全体を通じて、以下のルールを守ってください：

- **礼儀正しく、丁寧に、尊敬の念を持って、建設的に**評価プロセスに参加してください。コミュニティの健全性はこれにかかっています。
- 評価対象者（またはグループ）と協力して、作業の機能不全を特定してください。特定した問題について議論し、討論する時間を取ってください。
- プロジェクトの指示内容や機能の範囲について、同級生の理解に違いがある可能性を考慮してください。常にオープンマインドを保ち、可能な限り誠実に評価してください。教育は、ピア評価が真摯に行われた場合にのみ有効です。

## ガイドライン

- **学生またはグループのGitリポジトリにある作業のみを評価してください**。
- Gitリポジトリが学生またはグループに属していることを再確認してください。作業が関連するプロジェクトのものであることを確認し、「git clone」が空のフォルダで使用されていることも確認してください。
- 悪意のあるエイリアスが使用されて、公式リポジトリの内容以外のものを評価させられることがないよう注意深く確認してください。
- 驚きを避けるため、評価者と被評価者の両方が採点を促進するために使用される可能性のあるスクリプトを確認済みであることを慎重に確認してください。
- 評価する学生がまだその特定のプロジェクトを完了していない場合、その学生は防御を開始する前に課題全体を読むことが必須です。
- このスケールで利用可能なフラグを使用して、空のリポジトリ、動作しないプログラム、ノームエラー、チートなどを報告してください。これらの場合、採点は終了し、最終成績は0（チートの場合は-42）となります。ただし、チートを除いて、この失敗の原因となった問題を特定し、将来同じ間違いを繰り返すことを避けるために、作業について（完了していなくても）議論を続けることが推奨されます。
- 防御期間中は、**セグメンテーション違反、その他の予期しない、早期の、制御不能または予期しないプログラムの終了があってはならない**ことを覚えておいてください。そうでなければ最終成績は0になります。適切なフラグを使用してください。

**設定ファイルが存在する場合を除き、ファイルを編集する必要はありません。** ファイルを編集したい場合は、評価対象の学生と理由を明確にし、両方が同意していることを確認してください。

**メモリリークがないことも確認する必要があります。** ヒープに割り当てられたメモリは、実行終了前に適切に解放されなければなりません。コンピュータで利用可能なさまざまなツール（leaks、valgrind、e_fenceなど）を使用できます。メモリリークがある場合は、適切なフラグにチェックを入れてください。

## 必須部分

### コンパイル

- [ ] `make -n`を使用してコンパイルが`-Wall -Wextra -Werror`を使用しているかを確認（使用していない場合は「invalid compilation flags」を使用）
- [ ] minishellがエラーなしでコンパイル（そうでなければフラグを使用）
- [ ] makefileは再リンクしてはいけない

### シンプルコマンド & グローバル変数

- [ ] `/bin/ls`のような絶対パスでオプションなしのシンプルなコマンドを実行
- [ ] グローバル変数はいくつありますか？なぜですか？それが必須または論理的に感じる理由の具体例を示してください。
- [ ] 空のコマンドをテスト
- [ ] スペースまたはタブのみをテスト
- [ ] 何かがクラッシュする場合はクラッシュフラグを使用
- [ ] 何かが動作しない場合は不完全な作業フラグを使用

### 引数 & ヒストリ

- [ ] `/bin/ls`のような絶対パスでシンプルなコマンドを引数付きで実行（ただし引用符とダブル引用符なし）
- [ ] 異なるコマンドと引数で複数回繰り返す
- [ ] 何かがクラッシュする場合はクラッシュフラグを使用
- [ ] 何かが動作しない場合は不完全な作業フラグを使用

### echo

- [ ] 異なる引数で複数回繰り返す

- [ ] 何かが動作しない場合は不完全な作業フラグを使用

- [ ] 何かが動作しない場合は不完全な作業フラグを使用

### プロセスの戻り値

- [ ] `expr $? + $?`のようなものを試す

- [ ] 何かが動作しない場合は不完全な作業フラグを使用

- [ ] 引数なしのcatやgrepのようなブロッキングコマンド実行後にctrl-Dを試す
- [ ] 異なるコマンドで複数回繰り返す

**必須部分が優秀な場合にのみボーナスを確認します。** これは、必須部分を最初から最後まで完了し、ねじれたまたは悪い使用例の場合でもエラー管理が完璧でなければならないことを意味します。この防御中に必須部分ですべてのポイントを獲得しなかった場合、ボーナスは完全に無視されます。
